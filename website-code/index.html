üîß System Overview

Embedded Detection Unit

Our Real-Time Road Condition Mapper is composed of a device that samples acceleration using MPU6050 and detects potholes through sudden Z-axis spikes. Location data is gathered via GPS, and WiFiManager allows flexible network provisioning without hardcoded credentials. Detected events are uploaded to a cloud based database with geolocation metadata. A mobile app then pulls this data for visualization.

Cloud Data Management Layer

ThingSpeak stores pothole data (id, latitude, longitude, timestamp, status) with MATLAB Analysis scripts for automated cleanup, while Firebase handles user authentication.

üì± Mobile App Integration

Our mobile app displays pothole locations on a map interface, using real-time data uploaded on the cloud and route tracking. This allows drivers to anticipate hazards and choose safer paths. It can also notify users of new pothole reports based on proximity and movement direction.

Functional Requirements

The system supports the following functionalities:

  • Pothole Detection: The ESP32 processes MPU6050 data to identify potholes through acceleration spikes.
  • Location Tracking: The NEO-6M GPS module captures latitude, longitude, and timestamp for each pothole.
  • Data Upload: Pothole data is uploaded to ThingSpeak in JSON format over Wi-Fi.
  • Map Visualization: The Flutter app displays pothole locations as markers on Google Maps.
  • Route Planning: Users can input destinations and view routes with pothole counts, with the system suggesting the safest route.
  • Data Management: ThingSpeak stores structured data, with automated cleanup of records older than 30 days.

Hardware Components

The embedded unit includes:

  • ESP32: A microcontroller with built-in Wi-Fi, interfacing with MPU6050 via I2C and GPS via UART, uploading data to ThingSpeak via HTTPS.
  • MPU6050: A 6-axis accelerometer/gyroscope detecting vertical acceleration spikes for pothole identification.
  • NEO-6M GPS Module: Provides latitude, longitude, and timestamp for pothole locations.

Software Components

The software stack includes:

  • ThingSpeak: Stores pothole data and uses MATLAB Analysis scripts for automated cleanup of records older than 30 days.
  • Firebase Authentication: Secures user access to the Flutter app for pothole reporting and updates.
  • Flutter Mobile App: Retrieves data from ThingSpeak via REST APIs, displays potholes on Google Maps, and supports route planning and user settings.

Data Flow

The data flow is as follows:

  • The MPU6050 sends acceleration data and the NEO-6M GPS module sends location data to the ESP32, which formats it as JSON for upload to ThingSpeak.
  • ThingSpeak stores the data and provides it to the Flutter app via REST APIs.
  • The Flutter app authenticates users via Firebase, retrieves map data from Google Maps API, and displays pothole locations.
  • Road maintenance authorities query ThingSpeak for aggregated data, while system administrators manage ThingSpeak and Firebase configurations.

Limitations and Next Steps

The system faces several limitations:

  • Hardware Limitations: The ESP32's limited processing power restricts complex computations. Solution: Upgrade to ESP32-S3 or offload computations to ThingSpeak.
  • Wi-Fi Dependence: Unreliable in rural areas, risking data loss. Solution: Add offline buffering and a cellular module (e.g., SIM800L).
  • GPS Accuracy: The NEO-6M's 10-meter accuracy may be insufficient in urban areas. Solution: Use a high-precision GPS module like u-blox NEO-7M with RTK.
  • Flutter Compatibility: Limited to Android 8.0+ and iOS 12.0+. Solution: Optimize for older OS versions and offer a web interface.
  • User Distraction: App interactions may distract drivers. Solution: Implement voice-activated controls and a simplified UI.

Next Steps:

  • Implement pothole reporting and debounce features in the Flutter app.
  • Conduct field tests in urban, rural, and highway environments.
  • Add offline data buffering to the ESP32 for unreliable network areas.
  • Pilot the system with road maintenance authorities and fleet operators.
